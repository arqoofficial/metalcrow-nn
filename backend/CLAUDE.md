# Backend

## Gotchas

### Testing / conftest

- `tests/conftest.py`'s `db` fixture is session-scoped and autouse, shared across the whole test run (only the User table is cleaned up at teardown) — assert before/after deltas within a test, never global row counts. An uncaught FK-violation flush in one test poisons the shared session (`PendingRollbackError`) and cascades false failures into every later test in that file until the offending statement is fixed.
- The `db` fixture only deletes `User` at teardown, so `Document` and `LiteratureSearch`/`LiteraturePaper` rows accumulate across test runs — this pollutes data-sensitive tests like `test_wiki_documents_empty`; recreate the scratch DB between full-suite runs to avoid false negatives.
- Route handlers receive a request-scoped `SessionDep` — a *different* `Session` instance from the module-scoped `db` fixture tests build fixtures with. An ORM object built via `db` can't be `session.add()`-ed inside a monkeypatched service; re-fetch it with `session.get(...)` in the request-scoped session, or SQLAlchemy raises "already attached to session".
- `normal_user_token_headers` always authenticates as `settings.EMAIL_TEST_USER` — to seed owned rows for it, look the user up via `crud.get_user_by_email(db, settings.EMAIL_TEST_USER)` and lazily `crud.create_user` it if absent.
- Pattern for capturing Celery dispatch in tests: monkeypatch `tasks.celery_app.signature` with a small class/lambda recording `(name, args)` plus a no-op `apply_async()` (see `tests/api/routes/test_ingest.py::test_enqueue_uses_v5_parse_task`).
- Celery `Task` objects (post-`@celery_app.task`) are directly callable as plain functions — task bodies can run synchronously in tests with just the `db` fixture, no live Celery/Redis needed; `.apply_async` is a monkeypatchable instance attribute for asserting dispatch without a real broker.
- `tests/utils/storage.py::FakeStorage.open_document` returns an object exposing only `.stream(chunk_size)`, not `.read()` — matches the real `minio` client's `get_object()`; `b"".join(obj.stream(8192))` is the correct read pattern.
- `tests/utils/storage.py::FakeRedisClient` only implements `incr`/`expire` (built for rate-limit tests) — not reusable where `.set`/`delete` are needed; write a separate fake inline for those.
- 2 pre-existing, unrelated test failures on this branch: `test_login.py::test_recovery_password`, `test_users.py::test_create_user_new_email` — both `FileNotFoundError` for missing built email templates, not caused by app changes.
- `uv run ruff format` reformats multi-line function signatures and wraps long trailing inline comments onto their own parenthesized line — run it before final commit, not just `ruff check`.

### Alembic

- `alembic revision --autogenerate` also picks up pre-existing, unrelated type drift (e.g. `experiments.documents.processing_level`: VARCHAR in DB vs StrEnum default in the model) — strip unrelated `alter_column` calls out of any new migration's `upgrade()`/`downgrade()`.
- Autogenerated `op.drop_table()` for tables with `sa.Enum` columns does NOT emit `DROP TYPE` — add explicit `sa.Enum(name='...').drop(op.get_bind(), checkfirst=True)` calls in `downgrade()` after the `drop_table` calls, or `upgrade → downgrade → upgrade` fails the second time with `type "X" already exists`.

### Models / mypy

- FK columns to `experiments.documents.id` (e.g. `LiteraturePaper.document_id`) are real FKs, not bare UUID columns — tests must insert/commit an actual `Document` row first and reuse its id, not assign a random UUID, or the flush fails with `ForeignKeyViolation`.
- Strict mypy (no `sqlalchemy.ext.mypy.plugin` configured) rejects `update(Model).where(A, B)` (comma multi-condition) with "incompatible type bool" — chain `.where(A).where(B)` instead; still needs a targeted `# type: ignore[arg-type]` per call. Repo already has precedent for targeted `# type: ignore[...]` comments (e.g. `app/api/routes/ingest.py:146` for `.in_()`).
- `@celery_app.task(name=...)` triggers strict mypy's `untyped-decorator` (no `celery-types` stubs installed) — add `# type: ignore[untyped-decorator]` on the decorator line.

### Schemas

- No root `app/schemas/__init__.py` re-export needed for new schema classes — that module only partially re-exports each domain module already, so per-module `__all__` is the real contract.
- `backend/tests/schemas/` may not exist yet for a given feature — add its own empty `__init__.py` when creating the first schema-focused test file.

### Config / clients

- `app.core.config.settings` is a module-level singleton — `monkeypatch.setattr(settings, "X", ...)` works in tests as long as the consuming module does `from app.core.config import settings` (same object reference) rather than re-importing/caching the value at call time.
- Sidecar HTTP clients (`app/services/science_kg_client.py`, `app/services/ontology_client.py`, `app/services/llm.py`) use module-level `httpx.get`/`httpx.post` calls with `trust_env=False`, not a persistent `httpx.Client` — follow this precedent for new sidecar clients even if a task brief suggests otherwise.
- Newer sidecar-client tests use `respx` for httpx mocking; older ones (`test_ontology_client.py`) use `monkeypatch.setattr(httpx, "get"/"post", MagicMock())` — both styles coexist intentionally, not an inconsistency to "fix".
- `json.dumps()` escapes non-ASCII by default — don't round-trip an httpx request body through `json.loads` → `json.dumps` to substring-search for non-ASCII text (e.g. Cyrillic); assert against the decoded dict fields directly.

### litsearch feature (`app/services/litsearch*.py`, `app/worker/litsearch_*.py`, `app/api/routes/litsearch.py`)

- Turn-group model: a chat turn's litsearches are grouped by reusing `LiteratureSearch.followup_of` as a turn-group key — the FIRST search (anchor) has `followup_of=None`, members point at it; Phase B (`agent_continue`) receives the anchor id and derives members via `_turn_search_ids`; the panel route aggregates the same way. `followup_search_id` is a DIFFERENT column — don't conflate.
- `_turn_search_ids(session, anchor_id)` and `_dedup_by_doi(papers)` are module-level functions in `litsearch.py` — reused by `litsearch_tools.py` via `litsearch._turn_search_ids(...)` pattern; call them this way for future shared turn-grouping logic.
- `_dedup_by_doi` keeps first occurrence per DOI; `doi is None` papers are all kept (never dedup None against None). Phase-B listing `idx` and the read tool `idx` both index into the same `_dedup_by_doi(_turn_search_ids order)` list — they MUST stay in sync or the model reads a different paper than it named.
- `LoopOutcome.literature_search_id` is now a read-only `@property` (= `literature_search_ids[0]` or None); never assign to it — safe because only read-side call sites exist.
- Cap semantics: `LITSEARCH_MAX_SEARCHES` = successful (≥1 paper) searches; `LITSEARCH_MAX_SEARCH_ATTEMPTS` = total attempts; loop forces an abstract answer when either is hit.
- The best-effort member READING flip in `agent_continue` must be wrapped in try/except and sits before the watchdog try/finally — an unguarded failure there strands the anchor at READING forever.
- Extracted PDF fulltext is never persisted (only a `fulltext_chars` count is stored) — `synthesize` must re-download + re-extract each `ADDED` paper's bytes via `storage.open_document`/`pdf_text.extract_text`, same as reconcile's `_mark_fetched`; there's no shortcut to reuse the earlier extraction.
- Redis lazy-singleton client pattern to copy: `_client: redis.Redis | None = None` + `_get_client()` doing `redis.Redis.from_url(settings.REDIS_URL)` (see `app/services/rate_limit.py`); tests monkeypatch `_get_client`/`_get_redis_client` directly rather than mocking `redis.from_url`.
- redis-py's `.set(key, val, nx=True, ex=...)` returns `None` (falsy) on a failed NX acquire, not `False` — wrap in `bool(...)` when checking.
- `litsearch_client.fetch_sync(doi: str, ...)` requires a non-`None` `str`; `LiteraturePaper.doi` is `str | None` — callers must guard `if paper.doi is not None` before calling (mirrors the DOI-required guard already used for `fetch_async` in `start_search`).
- article-fetcher's `FetchRequest.doi` is a required field — papers with only `pdf_url` and no DOI stay `FetchStatus.PENDING` after `start_search` (not fetched), left for the monitor/reconcile task.
- `settings.LITSEARCH_FETCH_TIMEOUT` (`app/core/config.py`) is specifically for `fetch_sync`'s long inline PDF-fetch chain (OpenAlex OA/EuropePMC/Sci-Hub/STC) — don't reuse `LLM_TIMEOUT` for it.
- `app/api/routes/ingest.py::run_ingest` is the canonical mirror pattern for task-record + dispatch: create the task row (`status=QUEUED`, ...), `session.add`/`commit`/`refresh`, then call the `enqueue_*`/`tasks.*` dispatch as a separate step (not inside the same transaction).
- `ChatMessage.message_metadata` is a plain `Column(JSON)` (not `JSONB`) — filter the metadata dict in Python rather than use JSONB comparators for small per-session message counts.
- `litsearch.start_search` already persists the "abstracts" answer as an assistant `ChatMessage` (`message_metadata.litsearch_kind="abstracts"`, `search_id=str(search.id)`) internally — callers must not write a second assistant `ChatMessage` for that turn, or chat history gets a duplicate assistant turn.
- A chat message is linked to its literature search only via `message.message_metadata.search_id` (+ `litsearch_kind`), not a DB FK (see `app/api/routes/litsearch.py::get_search`).
- Worker invocation: `celery -A app.worker.litsearch_app:celery_app worker -Q litsearch` (mirrors worker-graph/etl/embed's own-app-module convention).
- `docker compose config --services` includes services merged in from `compose.override.yml` (e.g. `playwright`) that aren't in `compose.yml` itself — diff against `compose.yml`'s own service list, not the merged config output, when verifying a compose edit.
- Check `services/article-fetcher/app/storage.py`'s docstrings directly for its env var contract (`MINIO_ENDPOINT` needs a scheme, `MINIO_PUBLIC_ENDPOINT`, `MINIO_ACCESS_KEY`/`MINIO_SECRET_KEY`, `MINIO_BUCKET`) rather than guessing when wiring a new consumer.

### RU literature search (`literature_search_ru` tool)

- **RU papers are terminal from birth** — `litsearch_client.search_ru()` returns papers with `fetch_status=SKIPPED` and `fulltext_status=ADDED` (full text inline from Cyberleninka API); they never enter the download/fetch cascade. `reconcile`'s `_TERMINAL_FETCH_STATUSES` already includes `SKIPPED`, so RU-only turns don't stall awaiting `job_status` polls.
- **RU papers dedup by title, not DOI** — `_dedup_papers()` keys on `doi:` if present, else `title:` (namespaced), so DOI-less RU papers dedup by title within the same turn.
- **RU search client timeout is longer** — `search_ru()` uses 45s timeout (vs 10s for EN `search()`) because it page-fetches article text inline via Cyberleninka's `fetch_fulltext()`.

### Agent loop tool-call caps (`app/services/agent/loop.py`)

- **Per-tool caps must be enforced INSIDE the `for tc in result.tool_calls` loop** (per-call), not just at iteration start — LLM can emit multiple tool-calls in one response. Checking caps only per-iteration allows a batch to overshoot: caller gets `max_successful_by_tool={"literature_search_en": 1}`, model calls both `literature_search_en` and `literature_search_ru` in one response, both execute (overshoot). See loop.py's in-loop `attempts_cap_hit`, `tool_cap_hit`, `global_cap_hit` checks.
- **Every skipped tool-call still needs a matching `role:"tool"` message in the next request** or the LLM 400s with "no tool result for call_id". Skipped calls are still recorded in `outcome.tool_calls_made` (the NAME only, for accounting).
- **Tool name dispatch** — `Tool.name` (in `tools` list) MUST exactly match the schema's `function.name`; `run_loop` dispatches via `{t.name: t for t in tools}` keyed on the LLM's actual call name. Renaming `SEARCH_SCHEMA["function"]["name"]` without updating `Tool.name=...` silently breaks dispatch (LLM calls the new name, loop finds no matching tool).
- **Tool schema renames ripple widely** — renaming schema `function.name` requires updates in: `Tool.name=...` (the handler registration), `chat.py`'s `first_tool_choice=...` literal, system prompt text sent to LLM (e.g., `"вызови инструмент literature_search_en"`), and any test assertions on registered tool names. Always grep the whole backend for the OLD schema name before deleting it.
- **`outcome.tool_calls_made` records only NAMES of tool-calls that actually EXECUTED** (not cap-tripped/skipped ones) — matches "made" semantics and helps track tool usage vs. attempts.
