# Вики Tab — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a `Вики` chat mode that lets the model search and read a
read-only local corpus of 562 pre-parsed OKF markdown files via two new
in-process tools (`grep_okf`, `read_okf`), run in the SAME native
tool-calling loop already built for litsearch (`agent/loop.py::run_loop`).
One synchronous loop, no Phase B, no side panel, no ingest — a grounding
source only.

**Architecture:** `ChatMode.WIKI` is a new early-return branch in
`chat.py::answer_message`, parallel to `LITERATURE`'s but simpler: it calls
`run_loop` once with `tools=[grep_okf_tool, read_okf_tool]`,
`first_tool_choice="grep_okf"`, and persists/returns the result directly —
no Celery dispatch. `grep_okf`/`read_okf` are pure-Python handlers
(`re`+`os.walk`, no subprocess) rooted at a new `settings.WIKI_CORPUS_ROOT`,
bounded (files scanned, results, snippet size, lines/chars read) and
containment-checked against path traversal (reusing the `.resolve()` +
`.relative_to()` idiom already in `app/services/okf.py`). See
`docs/superpowers/specs/2026-07-04-wiki-tab-design.md` for the full design
and the naming-collision note (this is NOT the existing sidebar `Wiki` page).

**Tech Stack:** FastAPI + SQLModel, pytest, `Path`/`re`/`os.walk` (stdlib
only — no new dependency), React + TanStack Query frontend, `openapi-ts`
codegen, Playwright for e2e.

  `app/services/wiki_tools.py`, `app/core/config.py`, `compose.yml`,
  `compose.override.yml`, `.env.example`, and the frontend Chat page/tab

  сформирован."` text, persisted and returned. Never a template/empty answer
  masquerading as a real one.

  `read_okf` must reject any path that resolves outside `WIKI_CORPUS_ROOT`

  scanned/results returned/snippet size so a broad pattern cannot dump the
  corpus into model context.
- **Tests use a fixture corpus** (`tmp_path` + `monkeypatch.setattr(settings,
  "WIKI_CORPUS_ROOT", ...)`), never the real 562-doc corpus at
  `/home/claude/a2a-shared/nornickel-cleaned-corpus/extracted/RAW_DATA/`.

## Verified codebase facts (read before starting)

- `agent/loop.py::run_loop(session, chat_session_id, messages, tools, *,
  max_iters, first_tool_choice=None) -> LoopOutcome{final_text,
  tool_calls_made, literature_search_id, degraded}` already exists and is
  generic. **Wiki tool results must never include a `"search_id"` key** —
  `run_loop` interprets that key as a literature search id and calls
  `uuid.UUID(str(...))` on it, which would raise on any other kind of id/
  string. `grep_okf`/`read_okf`'s return shapes (§ Task 2) carry no such key.
- `Tool` (dataclass): `name: str`, `schema: dict` (OpenAI function-schema
  shape), `handler: Callable` invoked as
  `handler(session, chat_session_id, **arguments) -> dict`. Wiki tool
  handlers ignore `session`/`chat_session_id` (stateless filesystem reads,
  no DB writes) but must accept them positionally — same adapter pattern as
  `litsearch_tools.make_search_tool`'s inner `handler`.
- `chat.py::answer_message` (`backend/app/services/chat.py:334-447`) branches
  on `mode` with an early return for `ChatMode.LITERATURE` at line 349-355,
  **before** the `is_gap_click`/`AUTO` waterfall (line 357 onward). The new
  `ChatMode.WIKI` branch goes in the same early-return position, right after
  the `LITERATURE` block. Unlike `LITERATURE` (which is also offered inside
  `AUTO` via `_run_litsearch_phase_a(primed=False)`), `WIKI` is **never**
  entered from `AUTO` — it is an explicit-only mode, same as
  `ONTOLOGY`/`KNOWLEDGE_GRAPH` today.
- `app/services/okf.py::read_okf_markdown` (lines 16-26) is the established
  containment-check idiom in this codebase: resolve the root, join+resolve
  the candidate path, then `full_path.relative_to(root)` inside
  `try/except ValueError`. `read_okf` (Task 2) reuses this exact pattern
  against the new `WIKI_CORPUS_ROOT`, **not** `OKF_ROOT` — those are
  different corpora (design doc §0.1).
- `Claim` schema (`app/schemas/chat.py`) requires `experiment_ids:
  list[uuid.UUID]` (no default — must pass `[]` explicitly) and
  `confidence: ClaimConfidence`. `ChatMessageResponse` has `literature:
  LiteratureRef | None = None` and `subgraph: SubgraphResponse | None =
  None` — wiki mode leaves both at their `None` default (it has neither).
- `frontend/src/routes/_layout/chat.tsx`'s `modeOptions` array (lines 33-54)
  is hand-maintained (not derived from the backend enum);
  `frontend/src/lib/postChatMessage.ts`'s `ChatModeUsed` union (lines 30-34)
  and `chat.tsx`'s `modeUsedLabel`/`modeUsedVariant` maps (lines 56-71) are
  separately hand-maintained too — all three need the new `"wiki"` value.
  `frontend/src/client/types.gen.ts`'s `ChatMode` union (line 44) is
  **generated** from the backend OpenAPI schema — regenerate via
  `bash scripts/generate-client.sh` after Task 3 lands, or `ChatMode.WIKI`

- `compose.override.yml` is committed (auto-loaded by `docker compose` next
  to `compose.yml`) and currently has **no hardcoded absolute host paths** —
  every host-specific value is `${VAR:-default}`. The new corpus bind mount

- No frontend unit-test framework exists for `.tsx` files in this repo — only

```python
2026-07-04-wiki-tab-design.md §2.3/§2.4). grep_okf/read_okf search and read a
read-only local corpus of pre-parsed OKF markdown rooted at

from app.core.config import settings
from app.services.agent.loop import Tool

        "description": (

            "journals) for a regex/substring pattern. Returns matching file "

def _scan_corpus(root: Path, max_files: int) -> tuple[list[tuple[str, int, str]], bool]:
    """Walks `root` for `.md` files (never following symlinks OUT of the
    tree — `followlinks=False`), materializing every (rel_path, line_no,
    line) triple. Fine for a corpus of a few hundred small markdown files;
    a much larger corpus would want a streaming/early-exit rewrite instead
    of materializing every line up front."""
    lines: list[tuple[str, int, str]] = []
    if not root.is_dir():
        return lines, False
    files_scanned = 0
    for dirpath, _dirnames, filenames in os.walk(root, followlinks=False):
        for filename in sorted(filenames):
            if not filename.endswith(".md"):
                continue
            if files_scanned >= max_files:
                return lines, True
            files_scanned += 1
            file_path = Path(dirpath) / filename
            try:
                text = file_path.read_text(encoding="utf-8", errors="ignore")
            except OSError:
                logger.warning("wiki corpus: failed to read %s", file_path)
                continue
            rel_path = file_path.relative_to(root).as_posix()
            for line_no, line in enumerate(text.splitlines(), start=1):
                lines.append((rel_path, line_no, line))
    return lines, False


def grep_okf(pattern: str, max_results: int | None = None) -> dict[str, Any]:
    cap = settings.WIKI_GREP_MAX_RESULTS
    if max_results is not None:
        cap = min(max_results, settings.WIKI_GREP_MAX_RESULTS)
    try:
        regex = re.compile(pattern, re.IGNORECASE)
    except re.error as exc:
        return {"matches": [], "truncated": False, "error": f"invalid pattern: {exc}"}

    lines, files_truncated = _scan_corpus(settings.WIKI_CORPUS_ROOT, settings.WIKI_GREP_MAX_FILES)
    snippet_chars = settings.WIKI_GREP_SNIPPET_CHARS
    matches: list[dict[str, Any]] = []
    truncated = files_truncated
    for rel_path, line_no, line in lines:
        if regex.search(line):
            matches.append(
                {"path": rel_path, "line_no": line_no, "snippet": line.strip()[:snippet_chars]}
            )
            if len(matches) >= cap:
                truncated = True
                break
    return {"matches": matches, "truncated": truncated}


def read_okf(path: str, offset: int = 0, limit: int | None = None) -> dict[str, Any]:
    root = settings.WIKI_CORPUS_ROOT.resolve()
    line_limit = settings.WIKI_READ_MAX_LINES
    if limit is not None:
        line_limit = min(limit, settings.WIKI_READ_MAX_LINES)

    full_path = (root / path).resolve()
    try:
        full_path.relative_to(root)
    except ValueError:
        return {"error": "path escapes corpus root"}
    if not full_path.is_file():
        return {"error": "file not found"}

    text = full_path.read_text(encoding="utf-8", errors="ignore")
    all_lines = text.splitlines()
    selected = all_lines[offset : offset + line_limit]
    content = "\n".join(selected)
    truncated = (offset + line_limit) < len(all_lines)
    if len(content) > settings.WIKI_READ_CHAR_CAP:
        content = content[: settings.WIKI_READ_CHAR_CAP]
        truncated = True
    return {"path": path, "content": content, "truncated": truncated}


def make_grep_tool() -> Tool:
    def handler(session: Session, chat_session_id: uuid.UUID, **kwargs: Any) -> dict[str, Any]:
        return grep_okf(kwargs.get("pattern", ""), kwargs.get("max_results"))

    return Tool(name="grep_okf", schema=GREP_SCHEMA, handler=handler)


def make_read_tool() -> Tool:
    def handler(session: Session, chat_session_id: uuid.UUID, **kwargs: Any) -> dict[str, Any]:
        return read_okf(kwargs.get("path", ""), kwargs.get("offset", 0), kwargs.get("limit"))

    return Tool(name="read_okf", schema=READ_SCHEMA, handler=handler)
```

Note on `path.relative_to(root)` when `path` is an **absolute** string like
`/etc/passwd`: `root / "/etc/passwd"` — `pathlib`'s `/` operator treats an
absolute right-hand operand as replacing the left entirely (POSIX semantics),
so `full_path` becomes `/etc/passwd` itself, which then correctly fails
`relative_to(root)`. This is exactly why the containment check must run
**after** the join+resolve, not be skipped because "it looked relative" —
covered by `test_read_okf_rejects_absolute_path_outside_root` above.

```python
        session,
        chat_session_id,

        text=content,
        experiment_ids=[],
        confidence=ClaimConfidence.LOW,
        kind=ClaimKind.FACT,
    )
    session.add(
        ChatMessage(
            session_id=chat_session_id,
            role=ChatRole.ASSISTANT,
            content=content,
            message_metadata={"mode_used": mode_used},
        )
    )
    session.commit()
    return ChatMessageResponse(
        claims=[claim],
        summary=content,
        tools_used=["wiki"],
        session_id=chat_session_id,
        mode_used=mode_used,
```
