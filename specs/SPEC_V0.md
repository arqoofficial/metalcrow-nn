# Научный клубок — Техническая спецификация

> **Проект:** Knowledge Graph / поисково-аналитическая система для НИОКР  
> **Команда:** 5 человек  
> **Формат:** хакатон, time-boxed delivery  
> **Версия:** 1.0  

---

## 1. Постановка проблемы

### Боль

Исследователи Норникеля работают с большим объёмом неструктурированных данных: внутренние отчёты, протоколы экспериментов, справочники материалов и оборудования, реестры сотрудников и лабораторий. Эти данные разрозненны — хранятся в PDF, DOC, таблицах, каталогах — и не связаны между собой. Чтобы ответить на вопрос *«что уже делали по сплавам X при режиме Y и какой был эффект на свойство Z?»*, исследователь вынужден:

- Вручную перебирать десятки документов
- Полагаться на память коллег и субъективный опыт
- Не видеть пробелов в экспериментальном покрытии
- Не иметь прозрачной истории решений с источниками

### Пользователи

| Роль | Потребность |
|------|-------------|
| **R&D-инженер** | Быстро найти релевантные эксперименты по составу, режиму и свойству |
| **Руководитель лаборатории** | Видеть картину покрытия: какие эксперименты проведены, каких не хватает |
| **Аналитик** | Строить связи между материалами, условиями и результатами; генерировать отчёты с provenance |

## 2. Цели и метрики успеха

### Цели

1. Построить работающий онлайн-прототип, который принимает корпус документов и позволяет задавать вопросы в свободной форме
2. Связать сущности (материалы, эксперименты, свойства, режимы, оборудование, команды, выводы) в единое пространство знаний
3. Обеспечить прозрачность: каждый ответ содержит ссылку на первоисточник (provenance)
4. Выявлять пробелы — какие эксперименты не проводились, но логически следуют из имеющихся данных

### Метрики успеха (KPI)

| Метрика | Целевое значение | Способ измерения |
|---------|-------------------|-------------------|
| **Время ответа на вопрос** | < 15 секунд (онлайн-режим) | Замер от запроса до полного ответа |
| **Точность извлечения сущностей** | > 80% F1 на hold-out set | Ручная разметка 50–100 фрагментов |
| **Provenance coverage** | 100% утверждений со ссылкой | Автоматическая проверка наличия proof-ноды |
| **Полнота графа** | > 70% сущностей из корпуса связаны | Отношение связанных нод к изолированным |
| **Демонстрация online-добавления** | Успешная переиндексация на hold-out данных за < 3 мин | Живое демо на презентации |
| **Кастомная RAG-метрика** | Выше baseline cosine similarity | A/B-сравнение на тестовых запросах |

### Ключевые принципы

- **Микросервисная архитектура** — каждый компонент в отдельном Docker-контейнере с FastAPI и `/docs` эндпоинтом
- **Read/Write разделение** — онлайн-агент работает только на чтение; запись изолирована в offline write-агент
- **Полная переиндексация** вместо инкрементальных миграций — проще и надёжнее для хакатона (~2.5 мин)
- **Провенанс по умолчанию** — каждый факт хранит ссылку на PDF-источник (документ + страница/абзац)

### Мета-сущности

| Сущность | Описание |
|----------|----------|
| **Источник (Proof)** | Ссылка на конкретный документ: файл, страница, абзац. Привязана к каждому факту |
| **Тег/Тематика** | Классификационная метка (напр. «суперсплавы», «коррозия», «палладий») |
| **Вывод (Conclusion)** | Текстовая интерпретация результатов эксперимента с proof_ref |

### Эксперимент-центричная таблица (PostgreSQL)

Основная плоская таблица для быстрого поиска:

```sql
CREATE TABLE experiments (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    title           TEXT,
    material_name   TEXT NOT NULL,
    material_formula TEXT,
    material_smiles TEXT,
    regime_json     JSONB,          -- {temperature, pressure, time, medium, steps[]}
    property_name   TEXT NOT NULL,
    property_value  FLOAT,
    property_unit   TEXT,
    uncertainty     FLOAT,
    equipment_name  TEXT,
    lab_name        TEXT,
    researcher      TEXT,
    conclusion      TEXT,
    source_doc      TEXT NOT NULL,   -- путь к исходному документу
    source_page     INT,
    source_paragraph TEXT,
    tags            TEXT[],
    embedding       vector(768),    -- pgvector
    created_at      TIMESTAMPTZ DEFAULT now()
);

CREATE INDEX idx_experiments_embedding ON experiments
    USING ivfflat (embedding vector_cosine_ops) WITH (lists = 100);

CREATE INDEX idx_experiments_material ON experiments (material_name);
CREATE INDEX idx_experiments_property ON experiments (property_name);
CREATE INDEX idx_experiments_tags ON experiments USING GIN (tags);
```

### Граф (Neo4j) — проекция для визуализации

```cypher
(:Material)-[:USED_IN]->(:Experiment)
(:Experiment)-[:UNDER_REGIME]->(:Regime)
(:Experiment)-[:MEASURES]->(:Property)
(:Experiment)-[:PERFORMED_AT]->(:Lab)
(:Experiment)-[:BY]->(:Researcher)
(:Experiment)-[:ON_EQUIPMENT]->(:Equipment)
(:Experiment)-[:HAS_RESULT]->(:Result)
(:Experiment)-[:SOURCED_FROM]->(:Source)
(:Material)-[:RELATED_TO]->(:Material)
(:Experiment)-[:LEADS_TO]->(:Conclusion)
```

Типы рёбер должны быть **множественными и типизированными** (не один тип `RELATED`).

## 5. Ключевые функции

### 5.1. Связывание сущностей (Entity Linking)

- Извлечение именованных сущностей из текста (материалы, режимы, свойства)
- Нормализация: приведение синонимов к каноническим формам (Cu-Ni-сплав → купроникель)
- Дедупликация через кластеризацию эмбеддингов (не через LLM — плохо работает с химической номенклатурой)
- Каждая сущность хранит массив алиасов и proof-ссылок

### 5.2. Гибридный поиск (Hybrid Retrieval)

Трёхступенчатый поиск:

1. **BM25** — полнотекстовый поиск по ключевым словам (русский язык, стемминг)
2. **Векторный поиск** — pgvector cosine similarity по эмбеддингам запроса
3. **Кастомная метрика** — расстояние между сплавами на основе различий в составе и условиях синтеза (не косинус, а domain-specific distance)

Результаты объединяются (reciprocal rank fusion) и проходят **LLM-реранкинг**.

### 5.3. Граф-запросы (Graph Traversal)

- Multi-hop запросы: «Какие свойства измерялись на сплавах, содержащих Pd, при температурах выше 800°C?»
- Cypher-запросы генерируются LLM-агентом на лету или через шаблоны
- Визуализация подграфов в интерфейсе

### 5.4. Детекция пробелов (Gap Analysis)

- Построение тензора `Материал × Режим × Свойство`
- Тензорная декомпозиция / missing link prediction для выявления незаполненных ячеек
- Ранжирование пробелов по потенциальной ценности
- Визуализация: heatmap покрытия экспериментов

### 5.5. История решений и Provenance

- Каждый факт в системе имеет `proof_ref` — ссылку на документ, страницу и абзац
- Timeline: хронология экспериментов по конкретному материалу/свойству
- Цепочки выводов: от эксперимента к заключению

### 5.6. Wiki-страницы

- Автоматическая генерация Markdown-страниц по сущностям (материал, лаборатория, исследователь)
- Три режима отображения: текст (Markdown), таблица, граф
- Полнотекстовый поиск по wiki

### 5.7. Генерация перспективных экспериментов [БОНУС]

> Связано с побочной задачей «Фабрика гипотез»

- На основе анализа пробелов — предложение экспериментов, которые стоит провести
- Ранжирование по новизне, рисках и потенциальной ценности
- Байесовская матричная факторизация для оценки uncertainty

## 6. Технологический стек

### Основной стек

| Слой | Технология | Обоснование |
|------|-----------|-------------|
| **Парсинг PDF/DOC** | UniExtract / ColPali + Marker | Визуальные энкодеры для сложных таблиц и формул |
| **Извлечение сущностей** | LangExtract + spaCy (ru) | LLM-экстракция с grounding для свободного текста; spaCy для структурированных данных |
| **Эмбеддинги текста** | `intfloat/multilingual-e5-large` [ASSUMPTION] | Поддержка русского языка, 768-мерные вектора |
| **Эмбеддинги молекул** | MatBERT / MolFormer | Специализированные модели для материалов и SMILES |
| **Основная БД** | PostgreSQL 18 + pgvector | Быстрое чтение, SQL, векторный поиск, GIN-индексы |
| **Графовая БД** | Neo4j Community | Визуализация, multi-hop запросы, демо для жюри |
| **Полнотекстовый поиск** | PostgreSQL FTS (tsvector) | Не нужен отдельный Elasticsearch для MVP |
| **LLM-агент** | LangChain + OpenAI API / Anthropic | Tool-use архитектура с набором инструментов |
| **Backend** | FastAPI (Python 3.11+) | Async, автодокументация, типизация через Pydantic |
| **Task Queue** | Celery + Redis | Очереди для тяжёлых операций парсинга |
| **Frontend** | React + TypeScript | SPA с чатом, wiki, визуализацией графа |
| **Визуализация графа** | react-force-graph / D3.js [ASSUMPTION] | Интерактивные графы в браузере |
| **Контейнеризация** | Docker + Docker Compose | Каждый сервис = контейнер |
| **Хранилище файлов** | S3-совместимое (MinIO для локальной разработки) [ASSUMPTION] | Хранение исходных PDF |
| **GPU-инференс** | Fireworks AI / Selectel / Cloud.ru | Внешний API для тяжёлых моделей (UniExtract) |

### Модели и эмбеддинги

```
Текст (русский) ──▶ multilingual-e5-large ──▶ vector(768)
Молекулы (SMILES) ──▶ MolFormer ──▶ vector(512) [ASSUMPTION]
Таблицы/PDF ──▶ UniExtract ──▶ Markdown + JSON
NER (русский) ──▶ spaCy ru_core_news_lg + custom pipeline
LLM-реранкинг ──▶ GPT-4o / Claude 3.5 через API
```

## 7. Ingestion & ETL Pipeline

### Общий поток

```
Документы (PDF, DOC, XLSX, каталоги)
        │
        ▼
┌───────────────────────────────┐
│  1. Конвертация               │
│  PDF → Markdown (UniExtract)  │
│  DOC → Markdown (pandoc)      │
│  XLSX → CSV                   │
└───────────┬───────────────────┘
            │
            ▼
┌───────────────────────────────┐
│  2. Извлечение сущностей      │
│  • Структурированные данные:  │
│    детерминированный парсер   │
│  • Таблицы:                   │
│    UniExtract → JSON          │
│  • Свободный текст:           │
│    LangExtract (few-shot LLM) │
└───────────┬───────────────────┘
            │
            ▼
┌───────────────────────────────┐
│  3. Нормализация              │
│  • Канонические имена         │
│  • Единицы измерения          │
│  • Алиасы и синонимы          │
└───────────┬───────────────────┘
            │
            ▼
┌───────────────────────────────┐
│  4. Дедупликация              │
│  • Эмбеддинги сущностей      │
│  • Кластеризация (HDBSCAN)   │
│  • Не удалять дубли, а        │
│    связывать с оригиналом     │
└───────────┬───────────────────┘
            │
            ▼
┌───────────────────────────────┐
│  5. Загрузка                  │
│  • PostgreSQL (эксп. таблица) │
│  • Neo4j (граф)               │
│  • OKF Markdown (wiki)        │
│  • CSV (бэкап/обмен)          │
└───────────────────────────────┘
```

### Детали по парсерам

| Тип источника | Парсер | Выход |
|---------------|--------|-------|
| Структурированный каталог (таблицы с фиксированными колонками) | Детерминированный парсер (pandas + regex) | CSV с типизированными полями |
| Сложный PDF с таблицами и формулами | UniExtract / ColPali | Markdown + JSON с bounding boxes |
| Свободный текст (отчёты, статьи) | LangExtract с few-shot примерами | JSON с сущностями + grounding (ссылка на фрагмент) |
| Справочники материалов | spaCy NER + словарь синонимов | Нормализованные записи материалов |

### Стратегия обновления

- **Полная переиндексация** — CSV-файлы перезаписываются, скрипт пересоздаёт индексы (~2.5 мин)
- **Hold-out set** — часть данных откладывается для демонстрации online-добавления
- Нет инкрементальных миграций — сознательное решение для упрощения в рамках хакатона

## 8. API Design

### Сервисы и эндпоинты

Каждый микросервис — отдельный FastAPI-контейнер с автодокументацией на `/docs`.

### 8.1. Search Service

```
POST /api/v1/search
```

Гибридный поиск по корпусу.

**Request:**
```json
{
  "query": "эксперименты с палладиевыми сплавами при температуре выше 800°C",
  "filters": {
    "material": "Pd",
    "temperature_min": 800,
    "tags": ["суперсплавы"]
  },
  "search_mode": "hybrid",   // "bm25" | "vector" | "hybrid"
  "top_k": 20,
  "rerank": true
}
```

**Response:**
```json
{
  "results": [
    {
      "experiment_id": "uuid",
      "title": "...",
      "material": "Pd-Cu сплав",
      "regime": {"temperature": 850, "medium": "аргон"},

      "source": {
        "document": "report_03.pdf",
        "page": 12,
```

## 9. UI/UX Концепция

### Основные экраны

#### 9.1. Главный экран — Чат с ассистентом

- Полноэкранный чат в стиле ChatGPT
- Каждый ответ содержит блоки:
  - **Текст ответа** — сгенерированный LLM
  - **Источники (Provenance)** — кликабельные карточки с документом, страницей, цитатой
  - **Связанные сущности** — теги-чипы (материал, режим, свойство), по клику — переход на wiki
  - **Мини-граф** — визуализация подграфа релевантных сущностей (inline)
- Боковая панель: история сессий, сохранённые запросы

#### 9.2. Wiki-страница сущности

Три вкладки отображения:

| Вкладка | Содержимое |
|---------|-----------|
| **Текст** | Markdown-страница: описание, формула, свойства, ключевые эксперименты |
| **Таблица** | Все эксперименты с этой сущностью в табличной форме (сортировка, фильтры) |
| **Граф** | Интерактивная визуализация связей сущности (react-force-graph) |

#### 9.3. Граф-эксплорер

- Полноэкранная визуализация графа знаний
- Фильтры по типу сущности, дате, тегам
- Кликабельные ноды → popup с деталями + переход на wiki
- Поиск пути между двумя сущностями

#### 9.4. Аналитика пробелов

- Heatmap `Материал × Свойство` — цвет = количество экспериментов
- Пустые ячейки = пробелы → по клику — предложение эксперимента
- Фильтры по режиму, лаборатории, дате

#### 9.5. Загрузка документов

- Drag-and-drop зона для PDF/DOC
- Progress bar парсинга (задачи через Celery)
- Превью извлечённых сущностей перед добавлением в базу

### Дизайн-принципы

- **Темная тема** — привычна для инженеров [ASSUMPTION]

## 11. Риски и митигации

| Риск | Вероятность | Влияние | Митигация |
|------|-------------|---------|-----------|
| **UniExtract стоит дорого / медленно** | Высокая | Высокое | Заранее посчитать экономику; fallback на Marker + LangExtract |
| **Качество NER на русском** | Средняя | Высокое | Few-shot примеры; словарь синонимов; ручная валидация на hold-out |
| **Neo4j медленно читает** | Средняя | Среднее | Neo4j только для визуализации; основа — PostgreSQL |
| **Не успеваем за время хакатона** | Средняя | Критическое | Чёткий MVP (п. 12); декомпозиция на независимые сервисы |
| **Химические синонимы ломают дедупликацию** | Высокая | Среднее | Эмбеддинг-кластеризация вместо строкового сравнения; не удалять дубли, а связывать |
| **LLM галлюцинирует** | Средняя | Высокое | Read-only агент; провенанс на каждый факт; реранкинг с проверкой |
| **Безопасность (прошлый опыт: систему «положили»)** | Средняя | Высокое | Изоляция write-агента; rate limiting; валидация входных данных |
| **GPU-ресурсы недоступны** | Низкая | Среднее | Облачные API (Fireworks, Selectel); fallback на CPU-модели |
| **Данные хакатона нетипичны** | Средняя | Среднее | Универсальная онтология; hold-out set для адаптации |

## 12. Вне скоупа (Out of Scope)

Следующие функции **не входят** в MVP хакатона:

- **Инкрементальное обновление** — только полная переиндексация
- **Редактирование wiki** пользователями — только автогенерация
- **Мультитенантность** — один инстанс, одна общая база
- **Авторизация и управление правами** — open access в рамках демо
- **Мобильная версия** — только десктоп
- **Автоматическое дообучение моделей** — только pre-trained + few-shot
- **Интеграция с внешними системами** Норникеля (ERP, LIMS) — только загрузка файлов
- **Подключение внешних баз** (Materials Project, COD) — желательно, но не в MVP
- **Production-ready мониторинг** (Prometheus, Grafana) — только логирование
- **CI/CD pipeline** — ручной деплой через Docker Compose

## Приложение C. Глоссарий

| Термин | Определение |
|--------|-------------|
| **Provenance** | Происхождение факта — ссылка на конкретный документ, страницу и абзац |
| **OKF** | Open Knowledge Format — Markdown-файл с метаданными для wiki-страниц |
| **Entity Linking** | Связывание упоминания сущности в тексте с каноническим представлением в базе знаний |
| **Gap Analysis** | Выявление пробелов в экспериментальном покрытии пространства «Материал × Режим × Свойство» |
| **Hold-out set** | Данные, отложенные для демо online-добавления |
| **Reranking** | Повторное ранжирование результатов поиска с помощью более точной модели (LLM) |
| **Grounding** | Привязка сгенерированного текста к конкретным фактам из источников |
| **Few-shot** | Режим работы LLM с несколькими примерами в промпте |
| **SMILES** | Simplified Molecular Input Line Entry System — текстовое представление молекулярных структур |
