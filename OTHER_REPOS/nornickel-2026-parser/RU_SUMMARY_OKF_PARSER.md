# OKF_PARSER: краткое описание

`OKF_PARSER` — сервис конвейерной обработки документов в формат OKF.

## Что внутри

- HTTP API для загрузки, статуса, дерева файлов и запуска обработки.
- Воркеры по стадиям пайплайна (`raw -> docling_raw -> docling_clean00`) через очередь задач.
- Слой конфигурации, блокировок, путей и сервисов обработки.
- Админ-панель и служебные скрипты для перезапуска, переиндексации и диагностики.
- Набор unit/integration/infra тестов для проверки API, пайплайна и runtime-сценариев.

## Ключевые особенности

- Четкая слоистая архитектура: `presentation`, `services`, `workers`, `data`, `config`.
- Устойчивое разрешение путей: `shared_root` корректно вычисляется независимо от текущей директории запуска.
- Поддержка локального и Docker-режима (Compose, env-переменные, отдельные сервисы API/воркеров).
- Ориентация на эксплуатацию: метрики, health-check, lock-механизмы и безопасный ретрай задач.

## Архитектура

- API принимает запросы и ставит задачи в очередь.
- Воркеры асинхронно обрабатывают задачи по этапам пайплайна, разделяя ответственность по стадиям.
- Сервисный слой инкапсулирует бизнес-логику (резолв путей, статусы, переиндексация, статистика).
- Конфигурация централизована и единообразно применяется для API и воркеров.

## ML-архитектура

- ML-обработка встроена в pipeline-стадии, где документы проходят OCR/парсинг и нормализацию в OKF.
- Модельные и кэш-артефакты вынесены в отдельные директории/тома, что упрощает прогрев и переиспользование.
- Параметры ML-процесса управляются через конфигурацию и переменные окружения без изменения кода.

## Поддержка CPU и GPU

- Поддерживается запуск как на CPU, так и на GPU в зависимости от окружения.
- Режим вычислений контролируется через Docker/ENV-параметры (например, `TORCH_VARIANT`, `DOC_OCR_USE_GPU`).
- Для CPU и GPU предусмотрены одинаковые точки входа сервисов, что упрощает перенос между средами.

## Полный стек технологий (OKF_PARSER)

- Язык и рантайм: `Python 3.12+`.
- API и HTTP: `FastAPI`, `Uvicorn`, `python-multipart`.
- Валидация и конфиг: `Pydantic v2`, `PyYAML`, `python-dotenv`.
- Очереди и хранилище состояния: `Redis` (stage-очереди и лидер-лок основного сервиса).
- ML/OCR/парсинг документов: `Docling`, `EasyOCR`, `PyTorch`, `TorchVision`.
- Админ-интерфейс CLI/TUI: `Typer`, `Rich`.
- Логирование и наблюдаемость: `Loguru`, `prometheus-client`, `OpenTelemetry API/SDK`, опционально `Langfuse`.
- Тестирование: `pytest`, `fakeredis`, `httpx`.
- Инфраструктура и контейнеризация: `Docker`, `Docker Compose`, отдельные образы для API и воркеров.
- Управление зависимостями: `uv` (`pyproject.toml`, `uv.lock`), отдельный индекс CPU-колес PyTorch.

## Устройство очереди задач

- Используются две stage-очереди Redis (тип `LIST`):
  - `parser:jobs:raw2docling_raw`
  - `parser:jobs:docling_raw2docling_clean00`
- Формат задачи: `QueueJob` (`job_id`, `requested_path`, `resolved_path`, `stage`, `enforce`, `enqueued_at`), сериализация в JSON.
- Публикация: `RPUSH` в очередь нужной стадии.
- Потребление: `BLPOP` с таймаутом (polling без busy-wait).
- Надежность: ретраи Redis-операций с backoff (`with_retry`) и lock лидерства `parser:main:leader` для singleton-поведения `main`.
- Поток выполнения:
  1. API принимает запрос и кладет stage0-задачу.
  2. Worker `raw2docling_raw` обрабатывает документ и при успехе ставит stage1-задачу.
  3. Worker `docling_raw2docling_clean00` выполняет очистку/нормализацию и завершает pipeline.

## Прогрев моделей и ONNX перед использованием

- В `OKF_PARSER` основной прогрев реализован для `Docling` и `EasyOCR` через `scripts/preload_models.py`.
- Прогрев включает:
  - настройку путей к кэшу моделей (`MODEL_CACHE_ROOT`, `EASYOCR_MODULE_PATH`, `HF_HOME`, `TRANSFORMERS_CACHE`, `TORCH_HOME`, `XDG_CACHE_HOME`);
  - инициализацию `easyocr.Reader(...)` (материализация весов OCR);
  - пробную конвертацию минимального PDF через `Docling` (инициализация pipeline и зависимостей);
  - запись sentinel-файла `.preload_done`.
- В рабочем режиме можно включить строгую офлайн-проверку (`REQUIRE_PRELOADED_MODELS=1`): сервис стартует только если sentinel и кэши действительно доступны.
- GPU-режим OCR управляется `DOC_OCR_USE_GPU` (`auto|true|false`) с проверкой наличия CUDA.
- Важно: отдельный ONNX-прогрев в коде `OKF_PARSER` не реализован как самостоятельный этап; здесь прогреваются именно Docling/EasyOCR/PyTorch-кэши.
