# benchmark — автоматический бенчмарк чат-ассистента (RAG)

Прогоняет набор **competence-вопросов** через сквозной HTTP-контур чата при
поднятом стеке и оценивает ответы, чтобы можно было убедиться, что система
работает — не проверяя каждый раз вручную. Метрики отражают требования задачи:
корректность, **провенанс** (источники, цифровые значения, патенты),
маршрутизацию источника знаний, латентность и честность на неотвечаемых вопросах.

Бенчмарк ходит по тому же публичному API, что и фронтенд
(`POST /api/v1/chat/sessions/{id}/messages`, SSE), поэтому меряет систему целиком:
роутер режимов → онтология / knowledge-graph / гибридный поиск → синтез ответа.

## Быстрый старт

Нужен поднятый стек (`docker compose up -d` в корне `metalcrow/`). Раннер
использует те же зависимости, что и бэкенд (`httpx`, `pydantic`, `pyyaml`) —
на dev/prod они уже стоят в общем `.venv`, ставить нечего.

```bash
# из корня metalcrow/ (Git Bash / Linux)
.venv/Scripts/python.exe -m benchmark.run                 # весь датасет, режим auto
.venv/Scripts/python.exe -m benchmark.run --sample 5      # дымовой прогон 5 вопросов
.venv/Scripts/python.exe -m benchmark.run --offline       # только валидация датасета
```

```powershell
# PowerShell (Windows)
.\.venv\Scripts\python.exe -m benchmark.run
# либо обёртка (аргументы пробрасываются как есть):
.\benchmark\run.ps1 --sample 5
```

Учётка для логина берётся из `metalcrow/.env` (`FIRST_SUPERUSER` /
`FIRST_SUPERUSER_PASSWORD`); переопределяется через `BENCH_USERNAME` /
`BENCH_PASSWORD` или флаги.

## Что меряется

На каждый вопрос собирается ответ чата (`summary` + тексты `claims` + встроенные
цитаты) и считаются метрики (0..1):

| метрика | что проверяет |
|---|---|
| **keywords** | наличие ключевых методов/терминов из эталона (`must_include_any/all`), с терпимостью к русским словоформам |
| **numeric** | числовые факты из эталона (`expected_values`): точное значение, диапазон, граница `≤`/`≥` с допуском и единицей |
| **provenance** | источники (цитаты / `experiment_ids` / совпадение названий док-тов) + «много цифровых значений» + **патенты** (когда вопрос их ожидает) |
| **mode** | сработал ли ожидаемый источник знаний (`mode_used`) / инструмент онтологии (`tools_used`) — мягкое ожидание |
| **latency** | НФТ ТЗ «сложный запрос за 3–5 c» (цель настраивается `--latency-target`) |
| **honesty** | для неотвечаемых вопросов (`answerable: false`) — честное «данных нет» вместо выдуманного факта |

Итоговый балл вопроса — взвешенная сумма применимых метрик
(`correctness 0.5 · provenance 0.3 · mode 0.1 · latency 0.1`; неприменимые
выпадают, веса перенормируются). Для неотвечаемых балл = honesty. Веса и формула —
в [`scoring.py`](scoring.py). Итог прогона — взвешенное по `weight` среднее +
разрезы по категориям, метрикам и латентности.

## Датасет

[`data/questions.yaml`](data/questions.yaml) — список вопросов
(`CompetenceQuestion`, схема в [`models.py`](models.py)). Собирается из двух
источников скриптом [`scripts/build_dataset.py`](scripts/build_dataset.py):

- **`data/generated/grounded_raw.json`** — вопросы, сгенерированные и выверенные
  по документам корпуса (папка «Источники информации (parsed)»): каждый несёт
  реальные числовые значения, названия источников и, где есть, **номера патентов**.
- **`data/seed_questions.yaml`** — ручные: флагманы из ТЗ и от экспертов
  (обессоливание воды, циркуляция католита при электроэкстракции Ni, распределение
  МПГ штейн/шлак, подача сырья в печи), структурные (маршрутизация к инструментам
  онтологии: противоречия, пробелы, эксперты, lineage, сравнение практик, покрытие),
  контроли честности (вопросы вне корпуса) и англоязычные (мультиязычность ТЗ).

Пример записи:

```yaml
- id: water_injection-savmin-sulphate-removal
  lang: ru
  category: water_treatment
  question: 'Как работает процесс SAVMIN и до какой концентрации сульфата снижено
    его содержание при испытаниях на Stillfontein?'
  expected_mode: ontology
  expected_tools_any: [evidence]
  gold:
    must_include_any: [SAVMIN, эттрингит, гидроксид алюминия]
    expected_values:
      - {label: сульфат на входе, value: 800, op: '=', unit: мг/л}
      - {label: сульфат на выходе, value: 200, op: '<=', unit: мг/л}
    expected_sources: [Методы очистки шахтных вод]
    expects_patent: false
    min_numeric_values: 3
    min_citations: 1
  answerable: true
  difficulty: medium
  grounding_note: '...откуда взяты значения...'
```

Категории покрывают воду (очистка/обессоливание), электроэкстракцию, распределение
МПГ штейн/шлак, подачу сырья в печи, выщелачивание, хлорирование, флотацию,
обеднение шлаков, рафинирование/аффинаж, закладку/геомеханику, осаждение,
ТЭП проектов + структурные/контрольные/маршрутные. Всего 107 вопросов
(435 ожидаемых числовых значений, 14 патентных, 4 контроля честности, 6 en).

### Добавить или пересобрать вопросы

Правьте `seed_questions.yaml` (руками) и/или `grounded_raw.json`, затем:

```bash
.venv/Scripts/python.exe -m benchmark.scripts.build_dataset
```

Скрипт нормализует, проверяет схему и уникальность id, пишет `questions.yaml`.
Можно и редактировать `questions.yaml` напрямую — формат тот же. Проверить
валидность без сервиса: `python -m benchmark.run --offline`.

Регенерация grounded-набора с нуля (LLM-фан-аут по документам корпуса) описана
в [`data/generated/README.md`](data/generated/README.md).

## Режимы прогона

По умолчанию каждый вопрос шлётся в режиме `auto` (как реальный пользователь).
Другие варианты:

```bash
--modes auto,ontology,knowledge_graph   # матрица режимов (каждый вопрос × режим)
--use-expected-mode                     # слать вопрос в его expected_mode
--only-category water_treatment,structural
--only-id flagship-pgm-matte-slag-tz
--lang en                               # только англоязычные
--sample 10                             # первые N (дымовой прогон)
--judge                                 # включить LLM-судью (см. ниже)
--fail-under 0.6                        # ненулевой exit-code, если overall ниже (для CI)
--pass-threshold 0.6                    # порог «вопрос зачтён»
--latency-target 5.0                    # цель латентности, c
```

## Отчёты

В `results/` пишутся три файла на прогон (папка в `.gitignore`):

- `run_<ts>.json` — полный дамп (`RunReport`): все ответы, метрики, агрегаты;
- `report_<ts>.md` — таблицы по метрикам/категориям/вопросам;
- `report_<ts>.html` — автономный HTML-отчёт.

Плюс сводка в консоль. Раздел «Компоненты» показывает, какие сервисы доступны:
`ontology_kg`/`science_kg` — internal-only сайдкары (порт наружу не публикуется),
поэтому по умолчанию `None`; их фактическое участие видно в `tools_used`/`mode_used`
каждого ответа. Прямые пробы — `--ontology-url` / `--science-url`, если проброшены.

## LLM-судья (опционально)

Флаг `--judge` включает оценку ответа LLM-судьёй (фактическая корректность,
полнота, заземлённость на источники) поверх правил. Нужен OpenAI-совместимый
эндпоинт: `BENCH_JUDGE_BASE_URL`+`BENCH_JUDGE_API_KEY` (или `OPENAI_*` / `LLM_*`
из `.env`). Любая ошибка судьи → метрика просто не учитывается, прогон не падает.

## Тесты

```bash
.venv/Scripts/python.exe -m pytest benchmark/tests -q
```

Юнит-тесты метрик, свёртки и валидации датасета — без поднятого сервиса
(в т.ч. проверка, что поставляемый `questions.yaml` валиден и уникален по id).
Годится как гейт в CI: `pytest benchmark/tests` (оффлайн) + опционально
`benchmark.run --fail-under` против поднятого стека.

## Как читать результаты

Итоговый балл — это состояние **развёрнутой** системы, а не «правильность»
датасета. Низкий балл по grounded-категориям при пустом индексе/неполной загрузке
корпуса — ожидаем и информативен: по мере наполнения онтологии и hybrid_search
балл растёт. Структурные вопросы (маршрутизация к инструментам онтологии) и
контроли честности проверяют, что сам контур и харнесс исправны, независимо от
полноты данных. Разрез «По категориям» показывает, где система уже отвечает, а где
ещё нет; «Слабейшие вопросы» — что чинить в первую очередь.
